# MOVP Suite v0.1.0 — Supabase + Cloudflare Deploy Implementation Plan

> **For agentic workers:** This is an **operator-executed deployment runbook**, NOT a subagent-auto-executed plan. It provisions paid cloud resources, mints credentials, and takes outward-facing/irreversible actions — a human runs each task and confirms its gate before the next. Steps use checkbox (`- [ ]`) syntax for tracking. Do not auto-execute end-to-end.

**Goal:** Deploy the complete MOVP runtime to a dedicated Supabase Cloud project + Cloudflare (Norcal Crew), verify all four norcalOS integration surfaces, then tag `v0.1.0`.

**Architecture:** Two managed planes — Supabase Cloud owns DB/auth/edge (42 migrations, 9 edge functions, PATs), Cloudflare hosts the Astro SSR frontend on Workers + two R2 buckets. Periodic job-draining (which delivers norcalOS webhooks) is wired with `pg_cron` + `net.http_post` invoking the `flows`/`content-scheduler`/`segment-recompute` functions, service key in Vault. norcalOS integrates as an authenticated cross-project client (MCP + service PAT + GraphQL + HMAC webhooks).

**Tech Stack:** Supabase (Postgres/GoTrue/Deno edge), Supabase CLI `2.109.1`, Cloudflare Workers + R2, `wrangler ^4`, Astro 6 + `@astrojs/cloudflare`, pg_cron + pg_net.

**Spec:** `docs/superpowers/specs/2026-07-16-movp-v0.1-supabase-cloudflare-deploy-design.md` (committed `d5193c3`).

## Global Constraints

- **Supabase CLI = `2.109.1`** (matches `pnpm check:supabase-cli-pins`); `setup-cli@v2`. Deno `2.9.2`, pnpm `9.12.0`, Node `22`.
- **Migrations are forward-only and frozen** — never edit/regenerate a merged migration. All schema reaches prod via `supabase db push` only.
- **No secret is ever committed.** Service-role key, R2 secret key, `RESEND_API_KEY` live in Supabase secrets / Vault. **Public values** (project ref, anon key, GraphQL endpoint URL, workspace id) go in plain `vars`/literals — never masked secrets (they hide formatting bugs).
- **Verify every CLI flag** with `<cmd> <subcommand> --help` before running — flags differ per subcommand and per CLI version.
- **Content discipline:** deploy/verify output logs paths, refs, and reasons — never secret values or payload bodies.
- **Keep credentials out of argv and shell history.** Never inline a password/key as a literal in a command — pass it via an env var + `jq` building the JSON body piped through `curl --data @-` (so the secret is not a positional arg visible to `ps`). Prefix credential-bearing commands with a leading space (or `HISTCONTROL=ignorespace`) to skip history. Insert the Vault key from a **`psql` session**, never the dashboard SQL editor (which persists query history); if a secret is ever pasted somewhere persistent, rotate it.
- **Target Cloudflare account:** Norcal Crew (`ab0a68585c2155286627e8ea385e8709`); token from Keychain `cloudflare/norcal-crew/admin-token`.
- **Prod only** — no staging. **npm publish deferred** (not in this plan).
- **Resume after a partial failure:** most tasks are re-runnable; the non-idempotent ones (auth-user creation T3/T9, `create_workspace` T3, `vault.create_secret` T7) carry an inline re-run note — read it before repeating a task. Resume at the first task whose gate has not passed.

### Task 3: Bootstrap the owner user + first workspace

The suite ships **no `seed.sql`**. `public.create_workspace(p_name)` is `security definer` and reads `auth.uid()`, so it must be called **as an authenticated user** — a raw service-role/SQL call fails `not_authenticated`. This task creates the owner user, obtains a user session, and calls the RPC.

**Files:**
- Reference: `supabase/migrations/20260708000002_member_admin.sql:21` (`create_workspace`), `20260701000001_bootstrap_tenancy.sql`

**Interfaces:**
- Consumes: `PROJECT_URL`, `SERVICE_ROLE_KEY`, `ANON_KEY` (Task 1).
- Produces: `OWNER_USER_ID`, `WORKSPACE_ID` — consumed by Tasks 8, 9, 10.

- [ ] **Step 1: Create the owner auth user (admin API)**

Run (password via env var + `jq`/`--data @-` so it is not in argv; note the leading space):
```bash
 export OWNER_PASSWORD='<paste-strong-password>'
jq -n --arg pw "$OWNER_PASSWORD" '{email:"owner@yourorg.example",password:$pw,email_confirm:true}' \
| curl -sS -X POST "$PROJECT_URL/auth/v1/admin/users" \
    -H "apikey: $SERVICE_ROLE_KEY" -H "Authorization: Bearer $SERVICE_ROLE_KEY" \
    -H "Content-Type: application/json" --data @-
```
Expected: JSON with `"id": "<uuid>"`. **Record `OWNER_USER_ID`.** *(Re-run: if it returns `email_exists` / 422, the user already exists — skip and reuse the existing id via `GET $PROJECT_URL/auth/v1/admin/users`.)*

- [ ] **Step 2: Sign in as the owner to get a user JWT**

```bash
jq -n --arg pw "$OWNER_PASSWORD" '{email:"owner@yourorg.example",password:$pw}' \
| curl -sS -X POST "$PROJECT_URL/auth/v1/token?grant_type=password" \
    -H "apikey: $ANON_KEY" -H "Content-Type: application/json" --data @-
```
Expected: JSON with `"access_token": "<jwt>"`. **Record it as `OWNER_JWT`** (short-lived — use it immediately in Step 3).

- [ ] **Step 3: Create the workspace as the owner**

```bash
curl -sS -X POST "$PROJECT_URL/rest/v1/rpc/create_workspace" \
  -H "apikey: $ANON_KEY" -H "Authorization: Bearer $OWNER_JWT" \
  -H "Content-Type: application/json" \
  -d '{"p_name":"NorcalOS"}'
```
Expected: JSON row of the created `workspace` including `"id": "<uuid>"`. **Record `WORKSPACE_ID`.** *(Re-run: `create_workspace` is NOT idempotent — a second call makes a second workspace. If resuming, reuse the existing `WORKSPACE_ID` instead of re-calling: `curl "$PROJECT_URL/rest/v1/workspace?select=id,name" -H "apikey: $ANON_KEY" -H "Authorization: Bearer $OWNER_JWT"`.)*

- [ ] **Step 4: Gate — workspace + owner membership exist**

Run in SQL editor:
```sql
select w.id as workspace_id, m.role
from public.workspace w
join public.workspace_membership m on m.workspace_id = w.id
where w.id = '<WORKSPACE_ID>';
```
Expected: one row, `role = 'owner'`, `workspace_id = <WORKSPACE_ID>`.

### Task 7: Wire the drain schedulers (pg_cron + pg_net + Vault) — **load-bearing for webhooks**

Webhook delivery to norcalOS is **queued**: `emit_event` inserts a `kind='webhook'` job; only the `flows` function's `runFlowsWorker` performs the HMAC POST. Nothing invokes `flows` on a schedule out of the box. This task adds the three SQL scan crons (pure SQL) **and** the three HTTP drain crons (invoke the functions via `net.http_post`, service key from Vault). *(Alternative: Cloudflare Cron Triggers hitting the same URLs — see Risks. This plan uses the repo's documented pg_cron+pg_net pattern.)*

**Files:**
- Reference: `packages/flows/src/flows-worker.ts` (`runFlowsWorker`), the commented cron bodies in `20260701000009_task_lifecycle.sql:238`, `20260701000018_campaigns_bridge.sql:162`, `20260701000021_segmentation_recompute.sql:357`.

**Interfaces:**
- Consumes: `PROJECT_REF`, `PROJECT_URL`, `SERVICE_ROLE_KEY` (T1). Produces: 6 active `cron.job` rows; a draining `movp_jobs` pipeline.

- [ ] **Step 1: Store the service-role key in Vault (once)**

Run via **`psql`** (NOT the dashboard SQL editor — its query history persists the key). The key reaches psql on **stdin** (heredoc), never argv; `$DB_URL` is the project connection string (dashboard → Project Settings → Database):
```bash
 export SERVICE_ROLE_KEY='<paste-service-role-key>'
psql "$DB_URL" <<SQL
select vault.create_secret('$SERVICE_ROLE_KEY', 'movp_service_role_key', 'edge fn invoker key');
SQL
```
Expected: returns a uuid. *(Re-run: `vault.create_secret` errors on a duplicate name. If resuming, update instead: `select vault.update_secret((select id from vault.secrets where name='movp_service_role_key'), '$SERVICE_ROLE_KEY');`.)* If the key was ever pasted into a persistent surface, rotate it.

- [ ] **Step 2: Schedule the three HTTP drain workers**

In SQL editor (replace `<PROJECT_REF>`):
```sql
-- flows: notify + webhook delivery (every minute)
select cron.schedule('movp-flows-worker', '* * * * *', $$
  select net.http_post(
    url := 'https://<PROJECT_REF>.supabase.co/functions/v1/flows',
    headers := jsonb_build_object('Content-Type','application/json',
      'Authorization','Bearer ' || (select decrypted_secret from vault.decrypted_secrets where name='movp_service_role_key')),
    body := '{}'::jsonb) $$);

-- content scheduler: publish due schedules (every 5 minutes)
select cron.schedule('movp-content-scheduler', '*/5 * * * *', $$
  select net.http_post(
    url := 'https://<PROJECT_REF>.supabase.co/functions/v1/content-scheduler',
    headers := jsonb_build_object('Content-Type','application/json',
      'Authorization','Bearer ' || (select decrypted_secret from vault.decrypted_secrets where name='movp_service_role_key')),
    body := '{}'::jsonb) $$);

-- segment recompute drain (every minute)
select cron.schedule('movp-segment-recompute', '* * * * *', $$
  select net.http_post(
    url := 'https://<PROJECT_REF>.supabase.co/functions/v1/segment-recompute',
    headers := jsonb_build_object('Content-Type','application/json',
      'Authorization','Bearer ' || (select decrypted_secret from vault.decrypted_secrets where name='movp_service_role_key')),
    body := '{}'::jsonb) $$);
```
Expected: three integer job ids.

- [ ] **Step 3: Schedule the three pure-SQL scan crons (no secret needed)**

```sql
select cron.schedule('task-due-soon', '*/15 * * * *', $$ select public.emit_due_soon(); $$);
select cron.schedule('campaigns-scan', '*/15 * * * *', $$ select public.scan_campaigns(); $$);
select cron.schedule('segments-full-recompute', '0 * * * *', $$
  select public.enqueue_job('segment_recompute',
           id::text || ':full:' || to_char(date_trunc('hour', now()),'YYYYMMDDHH24'),
           jsonb_build_object('segment_id', id, 'mode','full'), workspace_id)
    from public.segment where mode='dynamic' and active=true; $$);
```
Expected: three integer job ids.

- [ ] **Step 4: Gate — 6 jobs scheduled + flows actually drains**

```sql
select jobname, schedule, active from cron.job order by jobname;
```
Expected: 6 active rows (`campaigns-scan`, `movp-content-scheduler`, `movp-flows-worker`, `movp-segment-recompute`, `segments-full-recompute`, `task-due-soon`).

Then prove the drain works end-to-end (not a silent no-op) — force one flows invocation and confirm it returns 200:
```bash
curl -sS -X POST "$PROJECT_URL/functions/v1/flows" \
  -H "Authorization: Bearer $SERVICE_ROLE_KEY" -H "Content-Type: application/json" -d '{}'
```
Expected: `200` with a JSON body reporting claimed/processed counts (0 is fine when the queue is empty). A `401` means the service key/Vault wiring is wrong — fix before trusting webhook delivery.

Finally prove pg_cron itself **fires on schedule** (a defined job that never executes would pass the checks above). Wait ~2 minutes, then:
```sql

from cron.job_run_details
where jobname = 'movp-flows-worker'
order by start_time desc limit 3;
```
Expected: at least one row with `status = 'succeeded'`. If there are no rows after 2+ minutes, pg_cron is not executing (extension/permission issue) — webhooks will silently never deliver; fix before proceeding. (Task 10 Step 3 exercises a real webhook end-to-end.)

## Risks & notes

- **Silent webhook non-delivery (Reliability, top risk).** If the `movp-flows-worker` cron or its Vault key is wrong, webhook jobs enqueue but never deliver — no error surfaces. Task 7 Step 4 + Task 10 Step 3 are the guards; do not skip either.
- **Scheduler alternative.** If you prefer the service key out of the DB, replace Task 7's HTTP crons with **Cloudflare Cron Triggers** — a scheduled Worker `POST`ing the three function URLs with the service key from CF secrets. The SQL scan crons (Step 3) still run in pg_cron either way.
- **Short-lived owner JWT.** `OWNER_JWT`/`SVC_JWT` expire quickly — mint and use them in the same session (Tasks 3, 9). On expiry, re-run the sign-in step.
- **Public-value discipline.** The four frontend `--var` values and the project ref are public; keep them as vars/literals, never `wrangler secret put` / masked CI secrets.
- **GraphQL root-field names** (Task 10 Step 2) — confirm via introspection against the deployed schema before asserting; the exact field may differ from `workspaces`.
- **Deferred (spec §9):** staging, SSO/shared human identity, npm publish, CI/CD auto-deploy, C7/C8.
