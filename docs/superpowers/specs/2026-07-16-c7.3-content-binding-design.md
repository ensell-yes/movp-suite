# C7.3 — Content Binding & Conflict Flow (design)

**Status:** approved (2026-07-16); ready for implementation plan. **Date:** 2026-07-16. **Branch:** `feat/c7-3-content-binding` (worktree, off merged `main` `f7d1357`).
**Depends on:** C7.2 `@movp/editor-sdk` (merged). **Feeds:** C7.6 delivery (doc-JSON published rendering).

**Execution:** two ordered parts, **C7.3a** (rich-text foundation) then **C7.3b** (editor binding
and conflict UX). C7.3 is complete only when both parts and their gates pass.

## 1. Goal & outcome

Bind `@movp/editor-sdk`'s `MovpEditor` to the real content-update revision mechanics on the CMS
editor page, so a rich-text field is edited in place, saved with optimistic concurrency, and a
concurrent-edit conflict is recovered **without losing the editor's in-progress draft**. Proven by a
genuine two-session Playwright conflict test.

**Success (gate):** two browser sessions edit the same item; session A saves (revision advances);
session B saves with the now-stale base revision → B receives a conflict; **B's draft text remains
present**; B refreshes the revision and re-saves; B's re-save persists and **preserves A's changes to
other fields**. A content type with **two** rich-text fields saves both sequentially in one session
without a self-conflict.

## 2. Decisions (resolved during brainstorming + review)

| # | Decision | Rationale |
|---|---|---|
| D1 | Rich-text field renders as a React island; saves via a server JSON endpoint | Exercises the SDK's real client-side save→conflict→refresh flow; keeps secrets server-side |
| D2 | Two-editor conflict e2e uses a **stateful** mock GraphQL server + two `browser.newContext()` | Runs in the fast `frontend-ux` job; no Supabase in CI |
| D3 | Rich-text stores **canonical ProseMirror doc-JSON**, canonicalized in **domain `prepare()`** | Single authoritative representation for ALL write surfaces (GraphQL/MCP/CLI/domain), not just the endpoint |
| D4 | One `RichTextFieldsIsland` coordinates all rich-text editors on the page over a shared revision | Independent islands would self-conflict when a type has ≥2 rich-text fields |
| D5 | Conflict recovery is **non-destructive**: refresh re-syncs the revision and leaves the draft | The SDK replaces the doc on `initialBody` change; auto-refresh would violate the "without data loss" gate |

## 3. The canonical rich-text invariant (addresses HIGH: representation not enforced)

**Invariant (authoritative):** for every `richtext` field, the value persisted to
`content_revision.data` is a **canonical doc-JSON string** — the byte output of the §5.2
`canonicalizeInnerJson` algorithm applied to a valid ProseMirror `doc`. This holds for **every** write
surface, because normalization happens in domain `prepare()`, below the GraphQL/MCP/CLI resolvers.

### 3.1 Shared canonical module (prevents editor↔domain drift)

`content_hash` is computed by the domain over the persisted value; the editor computes the same
canonical string when it encodes. If the two implementations diverge by one byte, hashes diverge and
idempotency/`content_hash` dedup break. Therefore a **single** implementation is shared.

Use a new client-safe leaf package `@movp/richtext` (no React, no TipTap, no server
imports) exporting:

```ts
// @movp/richtext
export function canonicalizeInnerJson(value: unknown): string   // moved from editor-sdk/src/canonical.ts
export function isDocShape(v: unknown): v is { type: 'doc'; content: unknown[] }
export function docToPlainText(doc: unknown): string             // concat text nodes, depth-first
export function normalizeToCanonicalDoc(value: unknown): string // see §3.2
```

`@movp/editor-sdk` re-exports `canonicalizeInnerJson` from `@movp/richtext` (back-compat; its public
surface test still passes) and `tipTapAdapter.encode` calls it. `@movp/domain` depends on
`@movp/richtext` for `normalizeToCanonicalDoc` + `docToPlainText`.

The alternative pure subpath export `@movp/editor-sdk/canonical` is rejected because it points the
dependency arrow domain→UI-SDK. `@movp/richtext` is the stable leaf both consumers depend on.

> **Gotcha (workerd/Deno):** `@movp/richtext` is imported by domain, which runs in Edge Functions.
> Every Edge entrypoint that imports `@movp/domain` must add `@movp/richtext` to its `deno.json`
> import map, and it must be in the `deno check` graph gate. Node typecheck will NOT catch a missing
> Deno import-map entry (see CLAUDE.md "Schema Productization").

### 3.2 `normalizeToCanonicalDoc` contract (state table)

| Input | Output |
|---|---|
| `''` (empty string) | canonical empty doc: `canonicalizeInnerJson({type:'doc',content:[]})` |
| valid doc-JSON string (parses + `isDocShape`) | `canonicalizeInnerJson(JSON.parse(input))` (idempotent on already-canonical input) |
| any other string (legacy plain text / HTML) | one paragraph of **escaped text**: `{type:'doc',content:[{type:'paragraph',content:[{type:'text',text:<input>}]}]}` canonicalized. HTML is treated as literal text (not parsed) in v1 — see Deferred. |
| non-string / null / undefined | throws `richtext_value_not_a_string` (the field is `z.string()`; a non-string is a caller bug) |

**Invariant:** `normalizeToCanonicalDoc(canonicalizeInnerJson(doc)) === canonicalizeInnerJson(doc)`
(idempotent on canonical input — the parity test in §9).

### 3.3 Domain `prepare()` change (atomic unit)

In `packages/domain/src/content.ts` `prepare()`, **before** `canonicalize`/`sha256Hex`: for each
`richtext` field present in `data`, replace `data[field.name] = normalizeToCanonicalDoc(data[field.name])`.
Then derive `search_body` for richtext fields as `docToPlainText(JSON.parse(normalizedValue))` instead
of `String(value)` (current `content.ts:170`).

- **Pre/post invariant:** persisted richtext value is canonical doc-JSON; `content_hash` is over the
  normalized data; `search_body` contains human-readable text, never `"type":"doc"` markup.
- **Atomic unit:** normalization + hash + search-text derivation are one pure step producing
  `{canonical, hash, searchText, searchBody}` — the RPC persists them together (§4).
- **Existing rows:** NOT bulk-migrated. Legacy plain-text values normalize on their next edit; reads
  tolerate them via the host normalizer (§6). No historical `search_body` rewrite.

### 3.4 Legacy-HTML rollout audit

Repository fixtures prove the common legacy case is plain text, but they do not prove deployed
databases contain no HTML. Before enabling C7.3 against an existing target, run this
content-disciplined classifier against **current revisions**. It emits counts only, never values:

```sql
with current_richtext_values as (
  select r.data ->> (field ->> 'name') as value
    from public.content_item ci
    join public.content_revision r on r.id = ci.current_revision_id
    join public.content_type ct on ct.id = ci.content_type_id
   cross join lateral jsonb_array_elements(ct.field_schema) as field
   where field ->> 'type' = 'richtext'
)
select count(*) filter (where value ~ '<[[:alpha:]][^>]*>') as html_like_values,
       count(*) as non_null_richtext_values
  from current_richtext_values
 where value is not null;
```

Expected for the repository seed/mock data: `html_like_values = 0`. A nonzero count on a deployed
target does not trigger implicit HTML parsing: v1 still converts that value to literal paragraph
text on its next edit, so visible tags are the documented fallback. Record the count before rollout
and plan explicit content cleanup if visible tags are unacceptable. Never print a matching value.

## 4. RPC retry ordering (addresses HIGH: identical retries false-conflict)

**Defect (confirmed `20260701000012_cms_content_rpcs.sql:69-79`):** `update_content` raises
`content_update_conflict` on a stale `p_expected_revision_id` **before** the hash-dedup shortcut. A
lost-response retry (first save committed, client never saw the new revision, retries same payload with
the stale base) therefore false-conflicts.

**Fix:** a **new forward-only migration** (`supabase/migrations/<ts>_update_content_hash_first.sql`) that
`create or replace`s `public.update_content` with the hash check **first**. Do NOT edit the frozen
`20260701000012` migration (CLAUDE.md migration discipline; guard: `pnpm test:forward-only-migrations`).

Reordered body (signature, SECURITY INVOKER, and all other logic unchanged):

```sql
-- after loading v_ws, v_parent, current_hash; after the not-found check:
if current_hash is not null and current_hash = p_content_hash then
  update public.content_item set search_text = p_search_text, search_body = p_search_body
   where id = p_item_id;
  select to_jsonb(ci) into result from public.content_item ci where ci.id = p_item_id;
  return result;                       -- idempotent success: no new revision, returns CURRENT revision
end if;
if p_expected_revision_id is not null and v_parent is distinct from p_expected_revision_id then
  raise exception 'content_update_conflict';   -- only for a DIFFERING payload on a stale base
end if;
-- ...unchanged: insert new content_revision, bump current_revision_id, update search fields...
```

- **Invariant:** an identical effective payload (same `content_hash`) is idempotent regardless of
  `p_expected_revision_id` staleness; a *differing* payload on a stale base conflicts. Key is derived
  over the **effective submitted** payload (matches the repo idempotency rule).
- **Atomic unit (unchanged):** revision insert + `current_revision_id` bump + `search_*` update in one
  statement-set; a conflict raise mutates nothing.

**pgTAP (extend `supabase/tests/cms_content_rpcs_test.sql` — tests are not frozen):**
1. stale `p_expected_revision_id` + identical hash → returns current revision, `content_revision` count
   unchanged, `current_revision_id` unchanged.
2. stale `p_expected_revision_id` + different hash → raises `content_update_conflict`.
3. the rejected call (case 2) changes neither the revision count nor `current_revision_id`.
4. (regression) matching base + different hash → new revision (existing behavior preserved).

## 5. Conflict recovery UX (addresses HIGH: refresh discards draft)

**Defect (confirmed `editor.tsx:34-38`):** the mount effect calls `editor.commands.setContent(...)` on
every `initialBody` change, so feeding refreshed content replaces the draft.

**Design — non-destructive recovery.** On `status === 'conflict'` the draft is never auto-replaced.
`ConflictSurface` offers two explicit actions:

- **"Refresh revision"** → `onRefresh()`: the coordinator (§6) fetches the item's **latest revision id**
  and updates its shared base revision. **`initialBody` is NOT changed → the draft stays.** The user
  then clicks **Save** to retry with the fresh base; the server merges the draft into the current
  `data`, so **other fields keep their latest values**. This is the "refresh path without data loss".
- **"Load latest field"** (clearly labeled destructive) → `onLoadLatest()`: the coordinator fetches the
  latest field value and sets `initialBody` → the SDK `setContent` **replaces the draft** with the
  winner's version. Used only when the user chooses to discard their edits.
- **GET failure** during either fetch → leave both the **draft and the conflict state intact**; surface
  an inline error; never clear the conflict on a failed refresh.

The coordinator owns a per-field refresh-feedback state:
`idle | refreshing | ready_to_retry | refresh_error`. Invoking **Refresh revision** sets `refreshing`;
a successful GET and shared-revision update set `ready_to_retry` and render
`<span role="status">Revision updated — Save to retry.</span>` adjacent to that editor. A failed GET
sets `refresh_error` and renders a safe inline `role="alert"`. A successful Save, successful **Load
latest field**, or a new conflict clears stale feedback to `idle`. The SDK may remain in its conflict
state so the draft and retry Save stay available; the coordinator's live region tells the user that
the refresh completed.

**SDK change (`@movp/editor-sdk`, TDD):** `MovpEditor` and `ConflictSurface` gain an optional
`onLoadLatest?(): void`; when provided it renders a second, destructive-styled button. `onRefresh`'s
documented semantics change to "re-sync revision, keep draft" (it no longer implies a reload). Copy:

> "This field changed since you opened it. **Refresh revision**, then Save to keep your version (other
> fields keep their latest). Or **Load latest field** to discard your changes and load the current
> version."

- **Invariant:** `status==='conflict'` ⇒ the editor document equals the user's draft until the user
  explicitly invokes `onLoadLatest`. A retry Save is enabled from the conflict state.
- **Atomic unit:** `onLoadLatest` is the ONLY path that mutates the draft on conflict.

## 6. Coordinator island (addresses HIGH: multi-richtext self-conflict)

`RichTextFieldsIsland` (new, `templates/frontend-astro/src/components/`) renders **all** rich-text
fields of the item and owns **one shared `currentRevisionId`** (a ref/state). Each child is a
`MovpEditor` for one field key.

- On any child's successful save, `onSaved(newRevisionId)` updates the shared `currentRevisionId`; every
  sibling's next save uses it → **no same-session self-conflict**.
- Each child save is independent and merged server-side by `fieldKey` (§7).
- Client-safe: no `@movp/domain`/`@movp/auth`/`@movp/graphql`/`@supabase`/`~/lib/server/*` imports.

**Invariant:** at most one authoritative `currentRevisionId` per item per island; a successful save by
any field advances it for all fields.

Page change (`content/[id].astro`): the rich-text field control(s) at `:327/:336` render
`<RichTextFieldsIsland client:load ... />` seeded with `{fieldKey, initialBody:
normalizeToCanonicalDoc(values[key]), itemId, initialRevisionId: current_revision_id}` per rich-text
field. Non-rich-text fields keep the existing form POST. Documented page rule (D5/F-B): a form save is a
full-page reload that re-mounts the island at the fresh revision, so **save rich-text before a form
save**; a `beforeunload` guard fires when any editor is dirty.

### 6.1 Exact dirty-state contract

The guard is backed by an SDK signal; it is not inferred from DOM events. Add this optional prop:

```ts
interface MovpEditorProps {
  // ...existing props...
  onDirtyChange?(dirty: boolean): void
}
```

`MovpEditor` keeps the last successfully loaded/saved canonical body as its baseline. Its TipTap
`onUpdate({ transaction })` handler returns immediately unless `transaction.docChanged`. On a
document change it transitions to dirty and emits `onDirtyChange(true)` **immediately**, so there is
no unguarded navigation window. It then resets one trailing 150 ms reconciliation timer. After the
quiet period, the timer encodes the current document once and compares it with the canonical
baseline; only that reconciliation can transition dirty back to false for undo-to-baseline. Thus a
typing burst does not perform O(doc) canonicalization per keystroke, while dirty protection is never
debounced. Initial/load-latest, successful Save, and unmount cancel the timer; Save uses the body it
already encoded for submission as the new baseline. State transitions are:

| Trigger | Baseline | Emitted dirty state |
|---|---|---|
| initial mount / changed `initialBody` (including explicit **Load latest field**) | canonical `initialBody` | `false` |
| first user edit / further typing burst | unchanged | `true` immediately; no repeated `true` callback |
| 150 ms reconciliation after edit/undo quiets | unchanged | exact canonical comparison result |
| successful save | submitted canonical body | `false` |
| conflict or save error | unchanged | remains `true` |
| successful **Refresh revision** | unchanged; `initialBody` is not changed | remains `true` |
| failed refresh/load-latest GET | unchanged | remains `true` |

Programmatic `setContent` for initial/load-latest state must not emit a transient dirty `true`; use
TipTap's non-emitting set-content path and set the baseline in the same effect. As with `onSaved`, a
throwing host dirty callback is contained and cannot break editor state.

The coordinator owns a `Set<fieldKey>` of dirty editors. Each child's callback adds/removes its key.
It installs exactly one `beforeunload` listener while the set is nonempty and removes the listener
when the set becomes empty or the island unmounts. The handler calls `event.preventDefault()` and
sets `event.returnValue = ''`; browser-native copy is used because modern browsers ignore custom
text. A clean editor never installs the guard.

**Tests:** the mounted SDK test uses fake timers to prove initial load is clean, a non-`docChanged`
transaction does not encode or emit, the first edit emits dirty immediately, a typing burst performs
one trailing encode rather than one per keystroke, undo-to-baseline becomes clean after
reconciliation, successful save is clean, and conflict/error remain dirty. A deterministic
coordinator unit test spies on
`window.addEventListener`/`removeEventListener`, proves exactly one `beforeunload` listener exists
while the dirty-key set is nonempty, dispatches the handler, and asserts the event is canceled. The
Playwright wiring test registers `page.once('dialog', handler)` **before** triggering the non-rich
form submission; the handler asserts `dialog.type() === 'beforeunload'` and dismisses the dialog so
the click can settle. The test then asserts URL and draft are unchanged. After saving rich text, the
test installs a dialog handler that records and dismisses any unexpected dialog, then awaits the same
form action's completed navigation; the completed navigation is the bound, so no timing sleep is
used. Assert the unexpected-dialog flag remains false, then remove that handler so it cannot leak
into later test cleanup.

## 7. Save/read endpoint (addresses MEDIUM: bounds + outcomes)

New Astro API route `templates/frontend-astro/src/pages/api/content/[id]/richtext.ts`. Server-side:
reads the `sb-access-token` httpOnly cookie and `readServerEnv()` **at call time** (never `process.env`;
never a service-role key — this route uses the **user's** bearer token so RLS applies).

**`POST` request** `{ fieldKey: string, body: string, expectedRevisionId: string }`:

1. **Resolve credentials at call time:** read the httpOnly cookie before consuming the request
   stream. A missing cookie returns `unauthorized` immediately. An expired/rejected token is detected
   by the combined authenticated read after the bounded parse and returns the same outcome; never
   retry the operation anonymously.
2. **Bound before buffer:** read the request stream with a hard **UTF-8 byte cap**
   (`MAX_BODY_BYTES = 262_144`, 256 KiB) BEFORE `JSON.parse`; over cap returns the `too_large` row and
   makes no upstream call.
   Do not call `await request.json()` unbounded (untrusted-I/O rule).
3. **Validate request shape:** `id` and `expectedRevisionId` are UUIDs; `fieldKey` is a nonempty string
   of at most 256 UTF-8 bytes; `body` parses as `isDocShape`. Malformed input returns the `validation`
   row with no upstream mutation. The endpoint does not canonicalize a valid document.
4. **One combined authenticated read:** call `CONTENT_ITEM_QUERY` exactly once with the user's token;
   it already returns `data`, `current_revision_id`, and `content_type.field_schema`. From that one
   response, reject an absent item, validate `fieldKey` names a `richtext` field, parse current `data`,
   and form the effective payload. Do not issue separate schema and data reads.
5. **Merge and write:** set `data[fieldKey] = body` with the validated document string unchanged, then
   call `updateContent(id, JSON.stringify(data), expectedRevisionId)`. Domain `prepare()` independently
   reloads and validates the type, canonicalizes rich text exactly once, and derives the effective
   payload/hash at the authoritative boundary. That defense is intentional, not a second endpoint
   read or a second normalization.

**Authoritative POST outcome table:**

| `outcome` event value | HTTP | Response body | Trigger |
|---|---:|---|---|
| `too_large` | 413 | `{status:'error', code:'body_too_large'}` | bounded reader crosses `MAX_BODY_BYTES` |
| `validation` | 422 | `{status:'error', code:'invalid_request'}` | malformed IDs/body/data or non-richtext `fieldKey` |
| `unauthorized` | 401 | `{status:'error', code:'auth_error'}` | missing/expired/rejected user token |
| `not_found` | 404 | `{status:'error', code:'not_found'}` | authenticated combined read returns no item |
| `saved` | 200 | `{status:'saved', revisionId:<new current_revision_id>}` | mutation succeeds |
| `conflict` | 409 | `{status:'conflict'}` | GraphQL error contains the real `content_update_conflict` code string |
| `error` | 500 | `{status:'error', code:'save_failed'}` | any other operational/upstream failure; message dropped |

This table is the only status mapping. The island narrows every response structurally; it passes
`saved`/`conflict` through to the SDK and maps every error variant to SDK `save_failed` after showing
the endpoint's safe code where useful.

**`GET ?fieldKey=<key>`** performs the same at-call-time authentication and one combined item read,
validates the field against `field_schema`, then returns
`{ body: <current normalized field value>, revisionId: <current_revision_id> }` for the coordinator's
refresh actions. It never logs or returns sibling field values.

**Observability (exactly one event per POST outcome).** Emit one content-disciplined structured event
per request with `{ event:'content.richtext_save', item_id?, field_key?, outcome:
'too_large'|'validation'|'unauthorized'|'not_found'|'saved'|'conflict'|'error', request_id,
latency_ms }`. `outcome` is
validated by value against that enum. **Never** log `body`, the token, or an unvalidated `field_key`
(only log `item_id` after UUID validation and `field_key` after it passes the schema check).

- **Discriminated union:** the response is one of `saved | conflict | error`; the HTTP status in the
  authoritative table is part of the contract, not only the JSON.
- **Idempotency:** a retried POST after an indeterminate failure re-attaches to the same base revision;
  the RPC hash-first rule (§4) makes an identical retry a success, not a conflict.

## 8. Stateful mock + e2e

**Mock (`templates/frontend-astro/tests/mock/graphql-mock.mjs`):** make `updateContent` stateful —

## 10. Deferred (not load-bearing for the C7.3 gate)

- **Published/delivery rendering of doc-JSON** → C7.6 (this spec only guarantees storage + search; the
  gate verifies existing delivery goldens do not regress, and defers doc-JSON *render* to C7.6).
- **Bulk migration of existing plain-text rich-text values** → none; they normalize on next edit.
- **HTML round-trip fidelity** — legacy HTML is stored as escaped literal text in v1, not parsed into
  marks. Parsing HTML→marks is deferred; the count-only pre-rollout audit and visible-tag fallback are
  defined in §3.4.
- **Realtime revision/presence** → C7.5.
- **Unified full-page island / multi-field single Save** → out of scope; dual-path documented (§6).
- **Rich-text in-place overlay on published host pages** → C7.4.

## 11. Security & boundary summary

- Endpoint uses the **user's** bearer token at call time (RLS-scoped); no service-role, no
  `process.env`, no anonymous read.
- `fieldKey` validated against the type schema before any merge → not an arbitrary field-write
  primitive; unknown keys are additionally stripped by domain zod.
- Request body bounded before buffering (413); structurally validated (422).
- Observability logs field **names**/**outcomes** only — never body, token, or unvalidated key.
- Client bundle: island + `@movp/richtext` are client-safe; the frontend boundary scan is extended to
  cover the new component directory and fails the build on a forbidden import.

## 12. Finalized review decisions

1. `@movp/richtext` is a new publishable leaf package; domain does not depend on the UI SDK.
2. `MAX_BODY_BYTES` is 262,144 bytes (256 KiB), measured over the UTF-8 request stream before
   buffering.
3. "Load latest field" ships in C7.3b as the only explicit draft-discard path.
4. `onDirtyChange?(dirty: boolean)` is the SDK mechanism behind the form-navigation guard.
5. Execution is split C7.3a→C7.3b; both parts are required before reporting C7.3 complete.
