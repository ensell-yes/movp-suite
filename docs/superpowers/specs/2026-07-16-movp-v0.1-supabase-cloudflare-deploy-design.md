# MOVP Suite v0.1.0 — Supabase + Cloudflare Deploy for norcalOS Integration (Design)

**Status:** design (brainstormed 2026-07-16) — pending user review, then implementation plan.
**Baseline:** `origin/main` @ `b6658bb` (PR #22). C1–C6 executed and reviewed ≥9.2; C7/C8 unbuilt.
**Scope:** first production deployment of the MOVP Suite and the `v0.1.0` release marker, wired so
norcalOS can integrate as an authenticated client.

---

## 1. Goal & intent

Deploy the **complete MOVP runtime** to managed infrastructure and cut the suite's first version
(`v0.1.0`), such that **norcalOS integrates across all four surfaces it needs**: agent/MCP calls,
data reads/writes (GraphQL), event webhooks, and a served frontend.

This is a v0.1 *integration deployment*, not a hardened multi-tenant GA. Success = every surface
round-trips against the live project and norcalOS can perform a real read and a real write.

## 2. What is being deployed (grounded in the repo)

- **DB:** 42 forward-only migrations. MOVP owns **52 `public.*` tables** (generic names like
  `task`, `comment`, `tag`, `note`, `campaign`, `segment`) plus **9 `movp_internal` tables** —
  61 `create table` statements in all — and **27 `reporting` views**; `reporting_bi` is created
  later by an operator RPC.
- **Edge functions — 9 deployable:** `mcp`, `graphql`, `flows`, `ingest`, `content-assets`,
  `content-scheduler`, `auth-exchange`, `index-embeddings`, `segment-recompute`. The `supabase/functions`
  dir also holds `_shared` (a shared lib — `ingest-bounds.ts`) and `_e2e` (an e2e smoke test); both are
  `_`-prefixed and **Supabase skips them on deploy**, so they are NOT prod functions.
- **Frontend:** `templates/frontend-astro` — Astro SSR (`output: 'server'`) on the
  `@astrojs/cloudflare` adapter. Its source config uses the Astro 6 adapter entrypoint; the build
  emits `dist/server/entry.mjs`, `dist/client/`, and the redirected deploy config. Astro's required
  `SESSION` KV binding is ID-less in source and Wrangler provisions it on first deploy.
- **Assets:** CMS assets are **Cloudflare R2** (the `content-assets` fn signs S3 requests against
  `<account>.r2.cloudflarestorage.com`). Storage is R2, not Supabase Storage — not a choice. One R2
  bucket is needed: `movp-assets`. The frontend has no runtime media-bucket consumer.

## 3. Decisions (this design)

| # | Decision | Choice | Rationale |
|---|---|---|---|
| D1 | Data/auth/edge plane | **Managed Supabase Cloud** | MOVP is Supabase-native (DB + GoTrue + Deno edge). |
| D2 | Frontend + storage plane | **Cloudflare (Norcal Crew account)** | Frontend already targets CF; R2 required for assets; colocated with norcal infra. |
| D3 | Project boundary | **Dedicated `movp-prod` Supabase project** | MOVP is architected as *sole owner* of `public` with 52 generic table names and **frozen forward-only migrations**. Co-owning norcalOS's DB would collide in `public` and violate the migration-freeze rule. norcalOS integrates cross-project over HTTPS instead. |
| D4 | Frontend host | **Cloudflare Workers via `wrangler`** | Current-gen SSR path; `wrangler` already in template devDeps. |
| D5 | Environments | **Prod only** | Fastest to a working v0.1 integration; one project, one Worker. |
| D6 | Version marker | **Tag `v0.1.0` after deploy verified** | Tag marks a *proven-deployable* commit; notes record the live refs. |
| D7 | norcalOS app-level auth | **Dedicated `norcalos-service` PAT** (user-scoped) | Reuses C3 exactly; all RLS applies as that principal. Human-identity SSO deferred (§9). |
| D8 | npm publish of `@movp/*` | **Deferred — service deploy only** | norcalOS consumes the live API, not npm packages. The OSS-publish track exists (§8c) but is out of scope for v0.1. |

**Rejected — shared Supabase project (co-owned DB):** MOVP's `CREATE TABLE public.task` (and 51
siblings) would collide with norcalOS's own `public` tables, and the migration freeze forbids
relocating MOVP off `public`. Isolation via a dedicated project is the only v0.1-safe option.

## 4. Architecture

```
Supabase Cloud: project "movp-prod"                 Cloudflare: Norcal Crew account
  Postgres  · 42 migrations                            Workers (SSR): Astro frontend
            · public.* (52) + movp_internal (9)              (@astrojs/cloudflare, unchanged)
            · reporting / reporting_bi                  R2: movp-assets
  Auth (GoTrue) + PATs (movp_internal)                   KV: auto-provisioned SESSION
                                                        + S3 API token → edge-fn secrets
  Edge Runtime · 9 fns                                        ▲
  pg_cron: due-soon · segments · scheduler                    │ frontend reads GRAPHQL_ENDPOINT
        ▲                                                     │  → movp-prod /functions/v1/graphql
        │  MCP HTTP + PAT · GraphQL · HMAC webhooks  ─────────┘
   norcalOS  (separate infra; authenticated cross-project client)
```

norcalOS never touches MOVP's database directly. It calls the **C3** (MCP/PAT) and **C5**
(ingest/webhook) surfaces MOVP already ships.

## 5. Deploy runbook (order is load-bearing)

Tooling: `supabase` CLI (per repo default; **CLI pinned `2.109.1`**, see repo CLAUDE.md) and
`wrangler` with the Norcal Crew Keychain token. Verify every subcommand flag with `--help` before
running (CLI-flag rule).

1. **Provision** `movp-prod` (record project ref + region). Enable extensions **`pgvector`,
   `pg_cron`, `pg_net`**.
2. **Schema:** `supabase link --project-ref <ref>` → `supabase db push`.
   *Expected:* 42 migrations applied; forward-only guard clean; `public`/`movp_internal`/`reporting`
   present; table count matches local.
3. **R2:** create bucket `movp-assets` (CMS) + an R2 **S3 API token** in Norcal Crew (Keychain token
   `cloudflare/norcal-crew/admin-token`). The frontend's `SESSION` KV is provisioned by Wrangler.
4. **Secrets:** `supabase secrets set …` (§6). No secret is committed.
5. **Functions:** deploy the **9** deployable functions (e.g. `supabase functions deploy --project-ref
   <ref>`; confirm the all-vs-per-function invocation with `supabase functions deploy --help` at plan
   time — the repo CLAUDE.md "no positional names" note is about `functions serve`, not `deploy`).
   *Expected:* `_shared` and `_e2e` are skipped (`_`-prefixed); confirm neither appears in the deployed
   list — a deployed `_e2e` would expose test surface.
6. **Cron (out-of-band):** run the `cron.schedule(...)` statements the migrations intentionally leave
   commented (`task-due-soon` `*/15 * * * *`; `segments-full-recompute` `0 * * * *`; content
   scheduler). The service-role key / function URL these jobs post to is stored in **Supabase Vault**,
   never inline.
7. **Seed:** create the initial workspace + owner user so RLS has a tenant.
8. **Frontend:** build `templates/frontend-astro` → `wrangler deploy` (Norcal Crew) with §6 env,
   pointed at `movp-prod`. Record the Worker URL.
9. **norcalOS wiring:** create `norcalos-service` MOVP user + workspace membership; mint its PAT;
   register norcalOS's inbound webhook endpoint with a per-webhook `secret`.
10. **Verify** (§7) → **tag `v0.1.0`** (§8).

## 6. Secrets & env (exactly what the code reads)

**movp-prod edge functions.** Supabase **auto-injects** `SUPABASE_URL`, `SUPABASE_ANON_KEY`,
`SUPABASE_SERVICE_ROLE_KEY` into deployed functions — do **not** set these. Set only:

| Secret | Value / source |
|---|---|
| `MOVP_JWT_ISSUER` | `<SUPABASE_URL>/auth/v1` |
| `R2_ACCOUNT_ID`, `R2_BUCKET`, `R2_ACCESS_KEY_ID`, `R2_SECRET_ACCESS_KEY` | R2 S3 token (step 3) |
| `RESEND_API_KEY` | Resend (notify path) |

*Not needed in prod (do not set):* `SUPABASE_JWT_ISSUER` — Supabase **rejects the reserved
`SUPABASE_` prefix** (the `secrets set` call errors), and it is redundant: the functions read
`Deno.env.get('MOVP_JWT_ISSUER') ?? Deno.env.get('SUPABASE_JWT_ISSUER')`, so `MOVP_JWT_ISSUER` wins.
And `GRAPHQL_URL` — read **only** by `_e2e/graphql_smoke.ts`, which is not deployed.

**Cloudflare Worker (frontend).** `SUPABASE_URL`, `SUPABASE_ANON_KEY`, `GRAPHQL_ENDPOINT`
(= `<SUPABASE_URL>/functions/v1/graphql`), `WORKSPACE_ID`.

**Webhook signing is per-webhook, not a global secret.** The flows worker HMACs each delivery with
the webhook row's own `secret` (`x-movp-signature: sha256=…`). norcalOS's secret is set when it
**registers** its endpoint (step 9), not as a deploy env var.

## 7. Verification gate (all must pass before tagging)

Each check must round-trip against `movp-prod` — and must assert **content**, not just HTTP 200
(RLS returns empty, not error, for an unauthenticated/wrong principal — a 200 can be a false pass):

- **MCP:** authed `tools/list` over `/functions/v1/mcp` with the service PAT → tool set returned.
- **GraphQL:** one workspace-scoped query returns **seeded rows** (not anonymous-empty).
- **Webhook:** a `task.*` / publish event delivers to a test endpoint with a **valid**
  `x-movp-signature`.
- **Assets:** presigned R2 upload **and** read-back succeed.
- **Frontend:** Worker URL loads `/login`; auth callback completes; one board renders.
- **norcalOS:** its PAT performs one real read **and** one real write.

## 8. Versioning & distribution (two orthogonal tracks)

Deploying the running instance and publishing to npm are **separate pipelines**. This v0.1 is the
**service-deploy track only**; the npm track is documented here but **deferred** (D8 / §9 / §8c).

### 8a. Service deploy → the `v0.1.0` git tag (in scope)

The norcalOS instance ships from **repo source** (`supabase db push` / `functions deploy` /
`wrangler deploy`) — **npm is not in this path**, and norcalOS integrates as an API client, not a
package consumer. Mark the release with an **annotated `v0.1.0` git tag** on the verified `main`
commit; tag message records the deployed Supabase project ref, Worker URL, deployed function digest,
and date. This is the whole "version" step for the deploy.

### 8b. Package versioning model (lockstep — reference)

The repo already enforces **lockstep fixed-version**, so there is nothing to invent:
`scripts/check-publishable-versions.mjs` hardcodes `EXPECTED_VERSION` and requires **all 12
publishable `@movp/*` packages** (`auth cli codegen core-schema domain flows graphql mcp
notifications obs platform search`) plus `create-movp` to sit at exactly that version; scaffolds pin
`^0.1.0`; `@movp/mcp-bridge` is intentionally **private at `0.0.0`** (excluded from the gate).

**Efficient bump (`0.1.0 → 0.2.0`)** — change one source of truth, let gates verify the rest:
1. Bump `EXPECTED_VERSION` in `check-publishable-versions.mjs`.
2. Rewrite all 12 manifests + `create-movp` + template `^` pins to match — via a small bump script
   that reads `EXPECTED_VERSION` and rewrites them with the owner-only atomic-write helpers (same
   discipline as codegen), so it is one command, not 15 hand-edits.
3. `pnpm check:publishable-versions` + `pnpm check:release-preflight` + the Verdaccio gate confirm
   consistency.

**Do not adopt Changesets** — it defaults to independent per-package versions and would fight the
fixed-version gate. Lockstep is the deliberate choice here.

### 8c. npm OSS-publish track (deferred — not v0.1)

The repo is already wired for it (publishable `publishConfig`, the `movp`-npm-org
`check:release-preflight`, and the C6d Verdaccio real-tarball boundary gate). When you choose to cut
the OSS release, the gated sequence is: Verdaccio dry-run → `MOVP_RELEASE_PREFLIGHT=1
check:release-preflight` (npm auth + `movp` org access) → `npm publish`. **Out of scope for this
deploy** because norcalOS consumes the live API, not npm packages.

## 9. Out of scope (v0.1)

- Staging environment (D5 — prod only).
- Shared human identity between norcalOS and MOVP (SSO / federated GoTrue). v0.1 uses a service PAT;
  end-users acting as themselves is a follow-up.
- Relocating MOVP off `public` (would require rewriting frozen migrations).
- **npm publish of `@movp/*`** (the OSS 0.1.0 release) — the track exists and is gated (§8c) but is
  deferred; norcalOS uses the live API. Cut it later without touching the deploy.
- C7 (inline editing) / C8 (RAG) — unbuilt.
- CI/CD auto-deploy pipeline — this runbook is manual for v0.1.

## 10. Named risks & mitigations (eight-dimension pass)

- **Correctness — false "empty is success":** RLS returns empty for the wrong principal, so a 200 is
  not proof. *Mitigation:* §7 checks assert seeded rows / signed payloads, never bare status.
- **Safety — public values vs secrets:** the Supabase project ref and anon key are public (ship to
  the browser); do not bury them in masked CI secrets (ci-deploy rule). R2 secret key, service-role
  key, `RESEND_API_KEY` are real secrets → Supabase secrets / Vault, never committed.
- **Reliability — pg_cron silent no-op:** a wrong Vault secret/URL makes schedulers fail quietly.
  *Mitigation:* verification includes one forced scheduler run + log check.
- **Reliability/Usability — PAT lifetime over long ops** (idempotency-cli-and-auth rule): norcalOS
  must attach the PAT **before** the first call and refresh-and-retry **once** on 401; token lifetime
  must exceed the operation window. Called out in the step-9 wiring task.
- **Observability — content-disciplined logs:** deploy/verify scripts log **paths, refs, and reasons**,
  never secret values or payload bodies.
- **Efficiency/Performance:** managed planes, no idle servers; frontend is CF-edge SSR. No hot-path
  regression introduced by the deploy itself.
- **Simplicity:** dedicated project + two managed planes is the least-moving-parts topology that
  satisfies all four integration surfaces; no self-hosted stack, no DB relocation.
